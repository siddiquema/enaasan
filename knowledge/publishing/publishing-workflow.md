# Publishing Workflow

## The core idea

Write once, in Markdown, with the metadata block from `architecture/metadata-standard.md`. Everything else — a PDF a student can download and print, a DOCX a reviewer can comment on, a Google Doc for collaborative editing, an HTML page embedded on the website — is a *rendering* of that one source, not a separate document that has to be kept in sync by hand.

## Conceptual pipeline

```
publications/<audience>/<document>/source.md   (canonical, versioned, front-matter metadata)
              │
              ├──► PDF renderer   ──► output/pdf/<document-id>.pdf
              ├──► DOCX renderer  ──► output/docx/<document-id>.docx
              ├──► Google Doc     ──► (external — see google-doc-standard.md)
              ├──► HTML renderer  ──► output/html/<document-id>.html
              └──► Website page   ──► (a separate, explicit task to embed into the live site)
```

No renderer currently exists. This diagram describes intent so that content written today doesn't have to be rewritten once a renderer does exist — as long as the Markdown follows `markdown-standard.md`.

## Why single-source matters here specifically

Enaasan already has one clear example of the problem this solves: the 90-Day Career Clarity Plan exists as a PDF generated by a standalone script, and its landing-page summary on `90-day-plan.html` is separately hand-written HTML describing the same content. If the plan's Week 4 guidance changes, both places need a manual, remembered update — nothing enforces that they stay consistent. A single Markdown source with defined renderers removes that risk structurally rather than relying on discipline.

## Process, once a renderer exists

1. Author writes/edits the Markdown source under `publications/<audience>/`, following the relevant `templates/` file and `standards/` requirements.
2. Metadata (`architecture/metadata-standard.md`) is filled in completely; `status` starts at `Draft`.
3. Content passes the review described in `standards/publication-standard.md`; `status` moves to `In Review` then `Published`.
4. Only published (or explicitly approved draft-preview) content is rendered to `output/`.
5. If the document is later revised, the version bumps per `versioning-standard.md`, and all output formats are re-rendered from the updated source — never patched individually.

## What doesn't change with this workflow

- Human review remains mandatory regardless of output format — a renderer producing a PDF doesn't skip the review step described in `standards/publication-standard.md`.
- The website itself (HTML pages, i18n system) is not restructured to consume this pipeline automatically. Embedding rendered knowledge-layer content into the live site is a deliberate, separate task each time, not an automatic side effect of publishing here.
