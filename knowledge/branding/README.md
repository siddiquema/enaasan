# branding/

## Purpose

Defines how enaasan sounds and looks, so that a Career Profile written this year and a Checklist written next year feel like they come from the same place. This is the layer everything else in `knowledge/` has to answer to — a technically correct publication that doesn't sound like enaasan has failed as surely as one with a factual error.

## Contents

| File | Covers |
|---|---|
| `brand-guidelines.md` | Mission, vision, core values — the "why" behind everything else |
| `writing-style.md` | Reading level, sentence construction, spelling conventions, accessibility |
| `tone-of-voice.md` | Voice and tone specifically — how to sound honest without sounding cold, plain without sounding condescending |
| `visual-language.md` | Colour, typography, illustration and photography style, document design principles |

## Naming Convention

Four fixed files. This folder is not expected to grow much — brand fundamentals should change rarely and deliberately, unlike templates or publications.

## Rules

1. Every document produced anywhere in `knowledge/` is checked against `writing-style.md` and `tone-of-voice.md` before it moves from `Draft` to `In Review` (see `standards/publication-standard.md`).
2. Visual assets created for any publication follow `visual-language.md`, including the specific rule about not using full dark backgrounds on printable first pages (see that file) — this exists precisely because a real publication ran into that ink-wastage problem.
3. These four files describe the brand as it already exists on the live site (`about.html`'s transparency section, the FAQ's tone, the 90-Day Plan's voice) — they are a description of an existing, working voice, not a new invention.

## Examples

- A Knowledge Card about CUET should read recognisably like the FAQ's answer to "Should I do B.Tech via JEE, or through CUET at a central university?" — same directness, same refusal to oversell or catastrophise, same India-specific framing.

## Future Expansion

- A `brand-assets-checklist.md` may be added once enough visual assets exist in `assets/` to need a single reference sheet (logo lockups, colour hex codes, approved font pairings) rather than prose description alone.
